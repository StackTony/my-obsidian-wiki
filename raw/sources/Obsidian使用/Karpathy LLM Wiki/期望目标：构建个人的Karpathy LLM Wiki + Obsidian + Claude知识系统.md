AI 驱动的"智能知识库"理念正在引发关注。传统笔记工具（如 OneNote）多用于被动记录，缺乏将知识主动提炼和组织的能力。与之相对，Karpathy 提出的 LLM Wiki 思路利用大语言模型持续"编译"原始资料，将散乱的信息汇集为结构化的 Markdown 个人知识库，并不断积累沉淀。

结合 **Obsidian** 作为本地文件型笔记环境和 **Anthropic Claude Code** 作为 AI 代理，我们可以构建一个自驱动的知识管理系统：LLM 自动读取新资料、生成摘要、更新百科式条目，用户仅需查询或增加资料即可。

本文从 **Why（为什么）**、**What（是什么）**、**How（怎么做）** 三个角度分析这一方案。

## 一、Why：为什么我们需要一种新的知识管理方式？

### 1.1 从我的OneNote说起

我使用OneNote已有多年。它是我的「第二大脑」，但这个大脑有个致命问题：**它只负责存储，不负责思考。**

我记录了上百篇工作思考、业务设计。但当我真正需要回答一个复杂问题时——比如总结「面向ADN L4目标态认知演进过程」——我发现：

- 这些知识是**碎片化**的，分散在不同的笔记本、分区、页面中
- 我需要**重新阅读**大量原始笔记才能找到答案
- 没有任何**知识之间的连接**，Notes之间是孤岛
- 最致命的是：**没有累积效应**。三个月前读的文章，我现在可能完全忘了它的存在

RAG（Retrieval-Augmented Generation）技术出现后，很多人以为解决了这个问题。但实际上，RAG只是让你_更快地找到_那些碎片，**知识并没有被真正整合**。LLM每次都要「重新发现」知识，从零开始拼图。

### 1.2 我们期望的知识管理与使用方式

我只需要寻找资料、探索发现、以及提出好问题，LLM 则负责所有的“苦力活”——总结、交叉引用、归档、记账，正是这些枯燥的工作让一个知识库随着时间推移变得真正有用。

在实际操作中，我会在屏幕一边打开 LLM Agent，另一边打开 **Obsidian（一款本地笔记软件）**。LLM 根据我们的对话修改文件，我则实时浏览结果——点击链接跳转、查看知识图谱（graph view）、阅读更新后的页面。  
**Obsidian 是开发工具（IDE），LLM 是程序员，而你的 Wiki 就是代码库。**


### 1.3 适应场景

1. 个人知识管理：追踪你的目标、健康、心理、自我提升——将日记、文章、播客笔记归档，随着时间推移构建一个结构化的“自我画像”。花费数周或数月深入研究一个主题——阅读论文、文章、报告，并逐步构建一个包含演进论点的全面 Wiki。
2. 团队知识共享：一个由 LLM 维护的内部 Wiki，数据来源可以是welink 聊天记录、会议记录、项目文档。

## 二、What：三个核心概念与它们的价值

### 2.1 LLM Wiki思路（编译器的理念）

Karpathy 认为，**LLM 应该是知识库的“编译器”而非简单的“检索器”**。

- **核心逻辑**：AI 不只是在需要时查一下资料，而是持续阅读原始素材（Raw Data），并将其实时编译成一个高度结构化、互联的 Markdown Wiki。
- **动态进化**：每当有新资料进入，Wiki 的实体页、概念页会由 AI 自动重写、补充并建立双向链接，形成知识的复利效应。

传统的RAG工作流：

用户提问 → 向量检索 → 拉取相关文档碎片 → LLM生成答案 → 结束  
（知识永远是"原材料"，每次都要重新加工）

LLM Wiki工作流：

摄入新文档 → LLM提取知识 → 更新wiki页面、建立链接、标记矛盾 → 知识被"编译"了  
用户提问 → 直接查询wiki（已有整合过的知识） → 答案更准确、关联更丰富

wiki是一个**持久的、可累积的、有生命力的智能**。它会随着你注入的每一篇文档而更加智慧，会自我维护跨引用，会标记新旧知识的矛盾。

### 2.2 Obsidian（AI 的持久化上下文层）

Obsidian是当前最流行的本地笔记+知识图谱工具。它的核心特点是：

- **本地 Markdown 存储**：Markdown 是 AI Agent 的原生接口。Obsidian 的本地文件结构让 Claude Code 能直接读写，无需复杂的向量数据库。
- **可视化图谱**：通过图谱，人类可以直观地观察 AI 整理出的知识结构，发现知识的枢纽与孤岛。
- **双向链接**系统（`[[页面名]]`）
- **丰富的插件生态**（Dataview, QuickAdd, Web Clipper等）
- **天然适合LLM操作**——它本质上就是一个文件夹里的文本文件

在Karpathy的设计中，Obsidian扮演着「IDE」的角色：LLM是程序员，负责写代码（维护wiki）；你作为使用者，是产品经理和审阅者，一边聊天一边浏览Obsidian中的实时变化。

### 2.3 Claude Code（具备执行权的代理）

这是目前最关键的一环。传统的网页版 AI 只能在对话框里交换文字，而 **Claude Code 具备操作文件系统的执行权**。

- **自主行动**：它能跨文件搜索、大规模重构笔记、自动修复死链，并根据预设的 Skill 执行复杂的维护任务。
- **深度集成文件系统**，能读、写、搜索、编辑本地文件
- **MCP（Model Context Protocol）支持**，可以调用外部工具
- **有记忆能力**，可以通过`CLAUDE.md`等文件理解项目规范
- **适合"vibe coding"**，即人与AI协作的轻量级编程模式

在LLM Wiki架构中，Claude Code就是那个负责「写wiki」的AI agent。它读你扔进来的原始文档，然后去Obsidian里更新、整合、链接所有相关页面。

### 2.4 三者融合的价值

|能力|贡献|
|---|---|
|**结构化知识积累**|LLM Wiki将碎片编译为连通的wiki|
|**可视化知识网络**|Obsidian的Graph View让你看清知识全貌|
|**AI原生工作流**|Claude Code作为「wiki程序员」自动化维护|
|**渐进式披露（Progressive Leakage）**|知识不是一次性倾倒，而是按需逐步展开|

### 2.5 渐进式披露：一个被低估的思路

「渐进式批漏」（Progressive Leakage），这个概念至关重要。他是skill发展的基石。是大模型时代必须遵从的规律，传统知识管理（无论OneNote还是RAG）都是**一次性暴露**：

- 要么全记（OneNote）
- 要么全检索（RAG返回top-k chunks）

但LLM Wiki的渐进式披露是：

- **第一次摄入**：LLM创建基础页面，标记「待深挖」
- **后续摄入**：更新、补充、连接，知识逐步深化
- **查询时**：LLM只调取与问题相关的那个知识切片，但背后有整个wiki作为支撑  
    这就像一个**活的知识有机体**：不是堆砌材料，而是持续生长。

**LLM Wiki是否比RAG更好？**

答案是：**在特定场景下，是的；但不是取代关系。**

- 当知识规模在**个人/小团队尺度**（几十到几百个源文档），wiki可以全部装入context window，**token效率更高**，且能建立跨文档的深层关联
- 当知识规模**超出context window**，或需要**精确溯源**，RAG仍然是工程上的最优解
- **最佳实践可能是混合架构**：稳定的知识用wiki，长期归档用RAG

Karpathy本人也提到：wiki适合「编译式」知识，而RAG是「检索式」。两者不是非此即彼，而是互补。

## 三、How：如何构建这三个能力

### 3.1 整体架构

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  Raw Source │────▶│  Claude Code │────▶│   Obsidian  │
│  (原始文档)  │     │   (LLM Agent)│     │   (Wiki)    │
└─────────────┘     └──────────────┘     └─────────────┘
      │                    │                    │
      │                    ▼                    │
      │            ┌─────────────┐              │
      │            │   Schema    │              │
      │            │ (CLAUDE.md) │              │
      │            └─────────────┘              │
      │                    │                    │
      ▼                    ▼                    ▼
  不可变文档            AI理解/编译          结构化知识网络
```

三层设计：

1. **Raw Sources**：你收集的原始文档（PDF、网页、笔记等），Immutable
2. **Wiki**：Claude Code生成的markdown文件集合，互相链接
3. **Schema**：`CLAUDE.md`告诉LLM如何维护这个wiki（入口、工作流、规范）

### 关键步骤：搭建Obsidian+Claude code环境（安装自行解决，Claudian插件可用），编写CLAUDE.md Schema，网上很多教程，就不累述

**Obsidian配置建议**：

- 开启「Internal links」和「Wiki-link auto-completion」
- 安装 **Dataview** 插件（用于动态查询）
- 安装**Claudian插件**
- 安装 **Web Clipper** 浏览器插件（快速抓取网页为markdown）
- 设置附件文件夹为 raw/assets/（本地保存图片）



回到最初的问题：
**Karpathy的LLM Wiki + Obsidian + Claude Code能产生什么化学反应？** 
答案是：
**一个会自己生长的知识伙伴。**而如果我们把目光放得更远一点——**这或许不只是知识管理的进化，而是AI Agent自我演进的起点。**个人知识库只是开胃菜。真正的大餐，是用wiki来构建一个会思考、会协作、会自我进化的Agent系统。接下来继续思考这套方法如何助力与ADN Claw的构建，从而支撑ADN Claw（NOE Mate）从单点能力走向整网级、自主演进的智能体系统。

Karpathy 自己的规模是**约 100 篇文章、40 万字**。
这个量级，靠这两个文件 + LLM 自己的能力，**完全不需要向量数据库。**
