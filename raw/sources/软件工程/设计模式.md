# 设计模式

软件设计是一门艺术，是对"变化"的辩证处理：发现变化，隔离变化，以不变应万变。

---

## 设计模式分类

| 分类 | 说明 |
|------|------|
| 创建模式 | 处理对象的创建过程 |
| 结构模式 | 处理类或对象的组合 |
| 行为模式 | 处理类或对象的交互和职责分配 |

### 设计模式的优点
- 面向接口编程
- 降低耦合性
- 增加灵活性

---

## 一、创建型模式（Creational Patterns）

### 1. Factory Method - 工厂方法

**定义**：定义一个用于创建对象的接口，让子类决定实例化哪一个类。Factory Method使一个类的实例化延迟到其子类。

**特点**：
- 客户类和工厂类分开
- 客户无须修改就可以接纳新产品
- 缺点是当产品修改时，工厂类也要做相应的修改

```java
public abstract class CarFactory {
    public abstract Car creator();
}

public class VW1Factory extends CarFactory {
    public Car creator() {
        return new Jetta();
    }
}
```

### 2. Abstract Factory - 抽象工厂

**定义**：提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。

```java
public abstract class CarFactory {
    public abstract Car creator();
    public abstract Truck creator(String s);
}
```

### 3. Builder - 建造模式

**定义**：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。

**特点**：建造模式可以强制实行一种分步骤进行的建造过程。

```java
public interface Builder {
    void buildPartA();  // 创建部件A
    void buildPartB();  // 创建部件B
    void buildPartC();  // 创建部件C
    Product getResult(); // 返回最终产品
}

public class Director {
    private Builder builder;
    public void construct() {
        builder.buildPartA();
        builder.buildPartB();
        builder.buildPartC();
    }
}
```

### 4. Prototype - 原型模式

**定义**：用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。

**特点**：
- 缺点是每一个类都必须配备一个克隆方法
- 将product和factory功能合二为一了

### 5. Singleton - 单例模式

**定义**：保证一个类仅有一个实例，并提供一个访问它的全局访问点。

**特点**：解决的主要是性能问题，而非耦合（变化）的问题。

```java
public class Singleton {
    private static Singleton instance = null;
    
    public static synchronized Singleton getInstance() {
        if (instance == null)
            instance = new Singleton();
        return instance;
    }
}
```

## 二、结构型模式（Structural Patterns）

### 1. Adapter - 适配器

**定义**：把一个类的接口变换成客户端所期待的另一种接口，从而使原本因接口原因不匹配而无法一起工作的两个类能够一起工作。

```java
public class PegAdapter implements IRoundPeg, ISquarePeg {
    private RoundPeg roundPeg;
    private SquarePeg squarePeg;
    
    public void insert(String str) {
        if (roundPeg != null) roundPeg.insertIntoHole(str);
        if (squarePeg != null) squarePeg.insert(str);
    }
}
```

### 2. Bridge - 桥梁模式

**定义**：将抽象部分与它的实现部分（行为）分离，使它们都可以独立地变化。

**特点**：解决2个方面的变化问题：抽象与实现（行为）。即一个类中多个方向的变化问题。

**遵循原则**：组合优于继承、面向接口编程、单一职责、Liskov替换等

### 3. Composite - 合成模式

**定义**：合成模式把部分与整体关系用树结构表示，使得用户对单对象和组合对象的使用具有一致性。

### 4. Decorator - 装饰模式

**定义**：动态给一个对象增加功能，这些功能可以再动态的撤销。

**特点**：
- 一个类可能有些额外的责任（加密、缓存、压缩等），是辅助主体业务的附着
- 以对客户端透明的方式扩展对象的功能，是继承关系的替代方案
- 提供比继承更多的灵活性
- 实际上是将Bridge中的抽象和实现合二为一了

```java
public class Decorator implements Work {
    private Work work;
    private ArrayList others = new ArrayList();
    
    public void insert() {
        otherMethod();
        work.insert();
    }
}
```

### 5. Facade - 外观模式

**定义**：外部与一个子系统的通信必须通过一个统一的门面对象进行，门面模式提供一个高层次的接口，使得子系统更易于使用。

**特点**：
- 每一个子系统只有一个门面类，只有一个实例（单例模式）
- 整个系统可以有多个门面类
- Example: JDBC

### 6. Flyweight - 享元模式

**定义**：以共享的方式高效的支持大量的细粒度对象。

**特点**：
- 采用类似于Hash表的方式，共享的思想
- 客户端不可以直接创建被共享的对象，应使用工厂对象负责创建
- 大幅度降低内存中对象的数量，主要解决OO性能问题

### 7. Proxy - 代理模式

**定义**：给某一个对象提供一个代理对象，并由代理对象控制对源对象的引用。

**特点**：
- 代理对象可以在客户和目标对象之间起到中介的作用
- 客户端分辨不出代理主题对象与真实主题对象
- 代理者对被代理者的功能进行扩展（OCP）

## 三、行为型模式（Behavioral Patterns）

### 1. Chain of Responsibility - 职责链

**定义**：很多对象由每一个对象对其下家的引用连接起来形成一条链。请求在这个链上传递，直到链上的某一个对象决定处理此请求。

**特点**：
- 客户并不知道链上的哪一个对象最终处理这个请求
- 处理者有两个选择：承担责任或者把责任推给下家
- 一个请求可以最终不被任何接收端对象所接受

### 2. Command - 命令模式

**定义**：把一个请求或者操作封装到一个对象中，将发出命令的责任和执行命令的责任分割开。

**特点**：
- 解耦服务的调用者与服务提供者
- 请求方只知道execute即可
- 系统支持命令的撤消

### 3. Interpreter - 解释器模式

**定义**：给定一个语言后，定义出其文法的一种表示，并提供一个解释器。

**特点**：
- 语言是指任何解释器对象能够解释的任何组合
- 需要定义一个代表文法的命令类的等级结构

### 4. Iterator - 迭代子模式

**定义**：可以顺序访问一个聚集中的元素而不必暴露聚集的内部表象。

**特点**：
- 将迭代逻辑封装到一个独立的子对象中
- 简化聚集的界面
- 每一个聚集对象可以有多个迭代子对象，状态彼此独立

### 5. Mediator - 调停者模式

**定义**：包装了一系列对象相互作用的方式，使得这些对象不必相互明显作用，从而可以松散偶合。

**特点**：
- 将多对多的相互作用转化为一对多的相互作用
- 架构级模式：Facade解决子系统间的问题，Mediator解决子系统内对象间复杂问题

### 6. Memento - 备忘录模式

**定义**：用一个备忘录对象存储另外一个对象内部状态的快照，在不破坏封装的条件下将对象状态外部化存储。

**用途**：可以在将来合适的时候把这个对象还原到存储起来的状态。

### 7. Observer - 观察者模式

**定义**：定义了一种一对多的依赖关系，让多个观察者对象同时监听某一个主题对象。主题对象在状态上发生变化时，会通知所有观察者对象。

```java
public class Product extends Observable {
    public void setName() {
        this.name = name;
        setChanged();
        notifyObservers(name);
    }
}

public class NameObserver implements Observer {
    public void update(Observable obj, Object arg) {
        // 处理名称变化
    }
}
```

### 8. State - 状态模式

**定义**：允许一个对象在其内部状态改变的时候改变行为，这个对象看上去象是改变了它的类一样。

**特点**：
- 把所研究的对象的行为包装在不同的状态对象里
- 每一个状态对象都属于一个抽象状态类的子类
- 管理类在不同状态下的行为

### 9. Strategy - 策略模式

**定义**：针对一组算法，将每一个算法封装到具有共同接口的独立的类中，使它们可以相互替换。

**特点**：
- 使得算法可以在不影响到客户端的情况下发生变化
- 把行为和环境分开
- 与State较像，类中封装算法，定义一组算法

### 10. Template Method - 模板方法模式

**定义**：准备一个抽象类，将部分逻辑以具体方法实现，然后声明一些抽象方法迫使子类实现剩余的逻辑。

**特点**：先制定一个顶级逻辑框架，将逻辑的细节留给具体的子类去实现。

```java
public abstract class Benchmark {
    public abstract void benchmark();
    
    public final long repeat(int count) {
        long startTime = System.currentTimeMillis();
        for (int i = 0; i < count; i++)
            benchmark();
        long stopTime = System.currentTimeMillis();
        return stopTime - startTime;
    }
}
```

### 11. Visitor - 访问者模式

**定义**：封装一些施加于某种数据结构元素之上的操作，使数据结构可以保持不变。

**特点**：
- 把数据结构和作用于结构上的操作之间的耦合解脱开
- 增加新操作很容易，就是增加一个新的访问者类
- 将有关的行为集中到一个访问者对象中
- 在已有类的层次上添加方法（OCP）
- 适用大的插件体系下
