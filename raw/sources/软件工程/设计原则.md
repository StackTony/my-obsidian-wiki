# 面向对象的设计原则

软件设计的艺术，是对"变化"的辩证处理：发现变化，隔离变化，以不变应万变。

## 六大设计原则

| 原则 | 英文名称 | 核心思想 |
|------|----------|----------|
| 面向接口编程 | Program To An Interface | 使用接口继承而非实现继承 |
| 依赖倒置 | Dependency Inversion | 高层和低层都依赖于抽象 |
| 组合优于继承 | Favor Composition | 优先使用组合获得更好的设计 |
| 单一职责 | Single Responsibility | 一个类只负责一个功能 |
| 开放-封闭 | Open-Closed | 对扩展开放，对修改封闭 |
| Liskov替换 | Liskov Substitution | 子类必须能替换父类 |

---

## 1. 面向接口编程（Program To An Interface, Not An Implementation）

### 核心定义
使用"接口继承"，而非"实现继承"。接口关注对象的概貌，将对象中"不变"的信息抽象出来，不涉及细节，因此是"稳定"的。

### 代码示例
```java
public interface IManeuverable {
    public void left();
    public void right();
    public void forward();
    public void reverse();
    public void climb();
    public void dive();
    public void setSpeed(double s);
    public double getSpeed();
}

public class Car implements IManeuverable { }
public class Boat implements IManeuverable { }
public class Submarine implements IManeuverable { }
```

### 面向接口的好处
- Client只需关注业务活动（如驾驶），不必关心具体实现
- 对象可以很容易被实现相同接口的另一个对象替换
- 对象间的连接不必硬绑定，增加灵活性
- 松散耦合，增加重用可能性

## 2. 依赖倒置原则（Dependency Inversion Principle）

### 核心定义
- 高层模块不应该依赖于低层模块，二者都应该依赖于抽象
- 依赖倒置原则使细节和具体实现都依赖于抽象
- 抽象的稳定性决定了系统的稳定性

---

## 3. 组合优于继承（Favor Composition Over Inheritance）

### 问题分析
继承在开发早期经常被过度使用：
- 子类组合数目几何级增长
- 子类类型编译时确定，不能在运行时动态改变

### 设计改进
- 优先使用组合可获得重用性与简单性更佳的设计
- 配合使用继承，扩充可用的组合类集，加大重用的范围

### 继承的正确使用场景
- 子类表达了"是一个父类的特殊类型"，而非"是一个由父类所扮演的角色"
- 子类的一个实例永远不需要转化为其它类的一个对象
- 子类是对其父类的职责进行扩展，而非重写或废除（LSP）

---

## 4. 单一职责原则（Single Responsibility Principle）

### 核心定义
- 所谓职责，可理解为功能，就是设计的类功能应该只有一个
- 职责也可以理解为引起变化的原因：当一个类中有两个以上的变化方向，会产生过多的变化点

### 解决方案
拆分变化！

---

## 5. 开放-封闭法则（Open For Extension, Yet Closed For Modification）

### 核心定义
OCP认为应该试图设计出永远也不需要改变的模块：
- 可以添加新代码来扩展系统的行为
- 不能对已有的代码进行修改

### 符合OCP的标准
- 模块的行为可以被扩展，以满足新的需求
- 模块的源代码是不允许进行改动的

### 重要观点
- 一个软件系统的所有模块不可能都满足OCP，但应努力最小化不满足OCP的模块数量
- OCP法则是OO设计的真正核心
- 符合该法则便意味着最高等级的复用性和可维护性

---

## 6. Liskov替换原则（Liskov Substitution Principle）

### 核心定义
使用指向基类（超类）的引用的函数，必须能够在不知道具体派生类（子类）对象类型的情况下使用它们。

### 理解要点
- LSP是根据对象的"多态"而得出的
- 在实现子类时必须要谨慎对待，以确保不会无意中违背了LSP

### LSP的保证
- LSP表明了ISA关系是与对象行为有关的
- 一个子类型不得具有比基类更多的限制
- 简单的做法是不要将基类中子类不需要的函数暴露给子类
