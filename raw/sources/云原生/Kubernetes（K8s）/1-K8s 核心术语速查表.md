## K8s 集群架构关键术语图示

### 控制平面 → 数据平面 流程
```
用户 YAML (声明式)
    ↓
【kube-apiserver】
    ├── 认证(Authentication) → 授权(Authorization) → 准入(Admission)
    └── 写入 etcd (期望状态)
    ↓
【调谐循环 (Reconciliation Loop)】
    ├── kube-controller-manager：Observe → Compare → Act
    │   ├── ReplicaSet Controller (副本管理)
    │   ├── Node Controller (节点健康)
    │   ├── Deployment Controller (滚动更新)
    │   └── Job/CronJob Controller (任务管理)
    └── kube-scheduler：Pod → Node 绑定
    ↓
【数据平面 (Worker Node)】
    ├── kubelet：Pod 生命周期管理 → 容器运行时(CRI)
    ├── kube-proxy：Service 负载均衡 → iptables/IPVS
    └── CNI 插件：Pod 网络分配与通信
    ↓
【外部接入层】
    ├── Ingress Controller (L7 路由)
    ├── NodePort / LoadBalancer (L4 暴露)
    └── Service Mesh (东西向 mTLS)
```

### K8s 网络四层模型
```
第一层：容器间通信 (Pod内 localhost)
    ↓
第二层：Pod间通信 (CNI: Flannel VXLAN / Calico BGP / Cilium eBPF)
    ↓
第三层：Service服务抽象 (ClusterIP + kube-proxy + CoreDNS)
    ↓
第四层：外部接入 (Ingress / NodePort / LoadBalancer)
```

### K8s 安全防护体系
```
【身份认证】  RBAC: ServiceAccount → Role → RoleBinding
【网络隔离】  NetworkPolicy: 默认Deny → 按需放行
【Pod安全】   PSS restricted: 非root + 禁提权 + 丢ALL Capabilities
【数据加密】  etcd EncryptionConfiguration → Secrets 加密存储
【API加固】   --anonymous-auth=false + AlwaysPullImages 准入
```



## 一、核心架构与哲学

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Kubernetes/K8s|容器编排平台|Google开源的自动化容器部署、扩展、运维和管理平台，云原生生态核心技术|
|Declarative API|声明式API|只声明"期望状态"(YAML)，由系统自动调谐到目标，而非命令式逐步操作|
|Imperative|命令式|逐步手动下达操作指令（如"加一个容器"），K8s 之前的传统运维方式|
|Reconciliation Loop|调谐循环|K8s 核心驱动机制：Observe(观察)→Compare(比较)→Act(行动)→Repeat，永不停止|
|Control Plane|控制平面|集群"大脑"，负责全局决策（调度、状态维护），包含 apiserver/etcd/scheduler/controller-manager|
|Data Plane|数据平面|集群"手脚"，真正运行容器化应用的工作节点(Worker Node)|
|Cluster|集群|一个完整的 K8s 运行单元，由控制平面+一组工作节点组成|

## 二、控制平面组件

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|kube-apiserver|API服务器|集群唯一入口，所有操作经此执行(RESTful API)，负责认证、授权、准入控制|
|etcd|分布式键值存储|集群"唯一事实来源"(Single Source of Truth)，存储所有期望状态和实际状态数据|
|Raft|一致性算法|etcd 使用的一致性协议：Leader选举+日志复制+Quorum多数决，保证强一致性|
|Quorum|法定人数|etcd 写入需大多数节点确认(如3台中2台、5台中3台)，允许少数节点宕机仍可用|
|kube-scheduler|调度器|为新创建的 Pod 选择最合适的 Node，基于资源需求、亲和性、反亲和性等策略|
|kube-controller-manager|控制器管理器|运行多种控制器的进程：ReplicaSet/Node/Deployment/Job/EndpointSlice/ServiceAccount 控制器|
|cloud-controller-manager|云控制器管理器|与云服务商集成的控制器(可选)，管理云负载均衡器、路由、存储等|

## 三、节点与运行时组件

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Node|工作节点|运行 Pod 的工作机器(物理机或虚拟机)，每个集群至少一个|
|kubelet|节点代理|每个 Node 上的"领班"，与 apiserver 通信，管理 Pod 生命周期（启动、停止、健康上报）|
|kube-proxy|网络代理|每个 Node 上的服务发现与负载均衡实现者，维护 iptables/IPVS 规则转发 Service 流量|
|Container Runtime|容器运行时|真正运行容器的软件，如 containerd、CRI-O，通过 CRI 接口与 kubelet 交互|
|CRI|容器运行时接口|kubelet 与容器运行时之间的标准接口协议|
|Pause Container|基础设施容器|每个 Pod 内的"根容器"，维持 Pod 的网络命名空间，其他容器共享其网络|

## 四、核心资源对象

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Pod|豆荚/最小部署单元|K8s 最小调度单位，一个或多个紧密相关容器的组合，共享网络和存储|
|Deployment|无状态部署|最常用控制器，管理无状态应用，支持滚动更新、扩缩容、回滚|
|ReplicaSet|副本集|Deployment 的底层控制器，确保指定数量的 Pod 副本始终运行|
|StatefulSet|有状态部署|管理有状态应用(如数据库)，保证 Pod 名称、网络标识、存储的唯一性和有序性|
|DaemonSet|守护进程集|在每个 Node 上运行一个 Pod 副本，用于日志收集、监控 Agent 等节点级任务|
|Job|一次性任务|运行一次性任务，保证任务成功完成后退出|
|CronJob|定时任务|定时周期性运行 Job，类似 Linux crontab|
|Replicas|副本数|期望运行的 Pod 实例数量，声明式管理的核心参数|

## 五、服务与网络

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Service|服务抽象|为 Pod 提供稳定的网络访问入口(ClusterIP)，解决 Pod 动态 IP 变化问题|
|ClusterIP|集群内部IP|Service 默认类型，仅集群内部可访问的虚拟 IP|
|NodePort|节点端口|在每个 Node 开放静态端口(30000-32767)，外部可通过"节点IP+端口"访问|
|LoadBalancer|负载均衡器|结合云服务商 LB 对外暴露服务，公有云常用|
|Ingress|七层路由入口|通过 Ingress Controller 实现 HTTP/HTTPS 域名路由，统一外部流量入口|
|Ingress Controller|入口控制器|真正干活的组件(如 Nginx/Traefik)，监听 Ingress 规则并动态配置路由|
|EndpointSlice|端点切片|Service 与 Pod 之间的链接映射，替代旧版 Endpoints 的可扩展方案|
|CoreDNS|集群DNS|K8s 默认 DNS 服务器，为 Service 提供 DNS 名称解析|
|kube-proxy iptables模式|iptables代理|基于 iptables 规则链实现 Service 负载均衡，O(n)复杂度，适合小集群|
|kube-proxy IPVS模式|IPVS代理|基于内核 IPVS 哈希表实现 Service 负载均衡，O(1)复杂度，大型集群标配|
|东西向流量|East-West Traffic|集群内部 Pod 之间的通信流量|
|南北向流量|North-South Traffic|集群外部(互联网)与集群内部的出入流量|

## 六、网络与 CNI 插件

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|CNI|容器网络接口|K8s 只定义网络标准，由插件(Flannel/Calico/Cilium等)实现 Pod IP分配和跨节点通信|
|Flannel|简单Overlay网络|VXLAN 随道模式，"包中包"封装，简单易用但有性能开销和 MTU 损耗|
|VXLAN|虚拟扩展局域网|Overlay 封装协议，将 Pod 数据包封装在 UDP 包中跨节点传输|
|Calico|BGP路由网络|纯三层路由，用 BGP 协议在 Node 间广播路由，无封装开销，性能接近裸金属|
|BGP|边界网关协议|支撑互联网路由的协议，Calico 用其在 Node 间宣告 Pod 路由信息|
|Felix|Calico代理|每个 Node 上运行的 Calico 组件，配置本地路由规则和 iptables/eBPF 策略|
|BIRD|Calico BGP客户端|每个 Node 上运行的 BGP 路由广播组件，将本地 Pod 路由宣告给其他 Node|
|Cilium|eBPF 网络方案|基于 eBPF 的下一代 CNI，可完全替代 kube-proxy，支持 L3-L7 网络策略|
|eBPF|扩展伯克利包过滤器|内核级"魔法系统"，安全注入小程序到内核钩子，原生机器码速度处理网络包|
|NetworkPolicy|网络策略|K8s 的 Pod 间防火墙规则，控制哪些 Pod 可以互相访问(L3/L4层面)|
|Overlay Network|覆盖网络|在物理网络之上构建虚拟网络的封装方案(如 VXLAN)|
|Weave Net|加密Overlay网络|支持 VXLAN/UDP，自动发现节点，提供加密通信的 CNI 插件|
|Kube-router|BGP路由方案|基于 BGP 的轻量 CNI，支持路由、防火墙、服务代理三种模式|
|Pod CIDR|Pod IP网段|每个 Node 分配的 Pod IP 地址范围(如 10.244.1.0/24)|
|MTU|最大传输单元|数据包最大尺寸，Overlay 封装会降低有效 MTU|

## 七、配置与存储

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|ConfigMap|配置映射|存储非敏感配置数据(环境变量、配置文件)，可挂载到 Pod 中|
|Secret|机密数据|存储敏感数据(密码、Token、证书)，Base64 编码(生产需启用加密)|
|Volume|存储卷|Pod 级存储，支持多种类型(本地/NFS/云存储)，随 Pod 生命周期|
|PersistentVolume(PV)|持久卷|集群级存储资源，由管理员创建，代表一块已存在的物理存储|
|PersistentVolumeClaim(PVC)|持久卷声明|应用开发者对存储的"租赁申请"("我需要10Gi冷库")，与 PV 自动绑定|
|CSI|容器存储接口|K8s 与存储厂商(AWS EBS/Ceph/NFS)之间的标准接口，类似 CNI 的存储版|
|StorageClass|存储类|定义存储"等级"(如 SSD/HDD)，PVC 可按等级动态 provisioning|
|Dynamic Provisioning|动态供给|PVC 创建时自动按 StorageClass 从存储后端创建 PV，无需管理员预创建|
|AccessMode|访问模式|PV/PVC 的访问方式：RWO(单节点读写)、RWX(多节点读写)、ROX(多节点只读)|
|EmptyDir|空目录卷|Pod 临时空目录，Pod 删除即消失，用于 Pod 内容器间临时数据共享|
|HostPath|主机路径卷|将 Node 文件系统目录挂载到 Pod，仅用于开发测试|

## 八、标签、选择器与调度

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Label|标签|键值对元数据(如 app=nginx, env=prod)，用于标识和分类资源|
|Label Selector|标签选择器|通过标签筛选资源的表达式，Deployment/Service 通过它关联 Pod|
|Annotation|注解|键值对元数据，用于存储非标识性信息(如构建版本、联系方式)|
|Node Affinity|节点亲和性|约束 Pod 只能调度到满足标签条件的 Node(如"必须有GPU")|
|Pod Affinity|Pod亲和性|约束 Pod 调度到与指定 Pod 相同的 Node/拓扑域(如"与前端在一起")|
|Pod Anti-Affinity|Pod反亲和性|约束 Pod 不调度到与指定 Pod 相同的 Node(如"副本分散在不同节点")|
|Taint|污点|Node 上的"排斥标记"，阻止不兼容 Pod 调度到此节点|
|Toleration|容忍|Pod 上的"豁免标记"，允许 Pod 调度到有对应 Taint 的 Node|
|Resource Request|资源请求|Pod 声明的最低资源需求(cpu/memory)，调度器据此选 Node|
|Resource Limit|资源上限|Pod 声明的最大资源限额(cpu/memory)，超出将被限速或终止|

## 十、安全与权限

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|RBAC|基于角色的访问控制|K8s 权限体系：Subject(ServiceAccount/User)→Role→RoleBinding/ClusterRoleBinding|
|Role|角色|命名空间内的权限规则集(定义可操作哪些资源、哪些动作)|
|ClusterRole|集群角色|集群级别的权限规则集(跨命名空间)|
|RoleBinding|角色绑定|命名空间内将 Role 绑定给 Subject|
|ClusterRoleBinding|集群角色绑定|集群级别将 ClusterRole 绑定给 Subject|
|ServiceAccount|服务账户|Pod 使用的身份标识，每个应用应使用独立 SA 绑定最小权限|
|PSS|Pod安全标准|替代已废弃 PSP，三个级别：privileged(宽松)/baseline(中等)/restricted(严格)|
|SecurityContext|安全上下文|Pod/Container 级安全配置：runAsNonRoot、allowPrivilegeEscalation、capabilities等|
|NetworkPolicy|网络策略|Pod间防火墙规则，支持 Ingress/Egress 流量控制，默认Deny是最佳实践|
|Admission Controller|准入控制器|API请求写入etcd前的拦截检查链(如 AlwaysPullImages/NodeRestriction)|
|mTLS|双向TLS认证|Service Mesh 中 Pod 间自动双向加密认证，实现零信任网络|
|Secret Encryption|Secret加密|etcd 中 Secrets 默认 base64 明文存储，需配置 EncryptionConfiguration 启用加密|
|Seccomp|安全计算模式|限制容器可调用的系统调用，RuntimeDefault 是推荐配置|
|Capabilities|Linux能力|细粒度权限控制，restricted 级要求 drop ALL 丢弃所有特权能力|

## 十一、服务网格与可观测性

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Service Mesh|服务网格|集群内部"东西向"流量治理层，在每个 Pod 注入 Sidecar 代理|
|Sidecar|边车代理|与应用容器同 Pod 部署的代理容器(如 Envoy)，劫持所有出入流量|
|Envoy|高性能代理|Istio 使用的数据平面代理，支持 L4/L7 负载均衡、mTLS、流量治理|
|Istio|服务网格平台|最流行的 Service Mesh，由 Istiod(控制面)+Envoy(数据面)组成|
|Istiod|Istio控制面|统一组件(Pilot+Citadel+Galley)，管理 Envoy 配置、证书分发、策略|
|Prometheus|监控系统|拉取模型(Pull)指标收集，K8S 服务发现自动发现监控目标|
|Grafana|可视化仪表盘|与 Prometheus 配合的监控图表展示平台|
|Loki|轻量日志系统|只索引标签不索引全文，存储成本极低，配合 Promtail 收集日志|
|Promtail|日志采集器|DaemonSet 部署在每个 Node，读取 /var/log/pods/ 容器日志推送给 Loki|
|DaemonSet|守护进程|确保每个 Node 上运行一个 Pod 副本，用于节点级采集/监控|
|EFK Stack|日志三件套|Elasticsearch(存储)+Fluentd(采集)+Kibana(查询)的日志方案|
|Jaeger|分布式追踪|收集 Trace ID + Span 绘制瀑布图，定位微服务调用链瓶颈|
|OpenTelemetry|可观测性标准|统一 Metrics/Logs/Traces 三叉戟的采集标准和 SDK|
|Trace ID|追踪号|请求进入集群时生成的唯一标识，在调用链中逐服务传递|
|Span|跨度|单个服务处理请求的工作记录，含 Trace ID 和时间戳|

## 十三、底层 Linux 内核术语

|术语|中文释义|简明解释|
| ---- | ---- | ---- |
|Namespace(Linux)|Linux命名空间|容器隔离基石：PID/MNT/NET/UTS/IPC/USER 等命名空间让进程"以为独占系统"|
|Cgroups|控制组|容器限制基石：控制进程组的 CPU/内存/IO 等资源配额(即 Pod resources.limits)|
|PID Namespace|进程命名空间|容器内进程列表隔离，容器只看到自己的进程(PID 1)|
|NET Namespace|网络命名空间|容器网络隔离，每个 Pod 有独立网络栈和 IP|
|MNT Namespace|挂载命名空间|容器文件系统隔离，每个容器有独立的挂载视图|
|iptables|内核规则链|Linux 内核包过滤框架，kube-proxy 用其实现 Service DNAT 转发|
|IPVS|内核负载均衡|Linux 内核 L4 负载均衡模块，哈希表查找 O(1)，大型集群优于 iptables|
|DNAT|目标地址转换|iptables/IPVS 将 Service ClusterIP 转换为 Pod 真实 IP 的核心操作|
