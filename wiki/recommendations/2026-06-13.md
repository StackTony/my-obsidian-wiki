## 九、核心术语表

### Harness Engineering 核心术语

| 术语 | 定义 |
|------|------|
| **Harness Engineering** | 驭缰工程，围绕 AI Agent 构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。核心哲学：人类掌舵，智能体执行 |
| **Agent = Model + Harness** | LangChain 提出的公式：智能体 = 大模型 + 驭缰系统。Harness 是模型之外的一切（工具、记忆、护栏、验证、编排） |
| **Feedforward** | 前馈控制，预防性约束——在 Agent 执行之前引导正确方向（AGENTS.md、类型系统、架构规则、Linter） |
| **Feedback** | 反馈控制，事后自修正——在 Agent 执行之后检测偏差并注入修正信号（测试、Hook、LLM-as-judge） |
| **Computational Sensor** | 计算型传感器，确定性反馈（Linter、类型检查、单元测试）——成功时沉默，失败时精确报错 |
| **Inferential Sensor** | 推断型传感器，非确定性反馈（LLM-as-judge、代码审查Agent）——需要概率性置信度 |
| **PEV Pattern** | Plan-Execute-Verify 三阶段架构模式：先规划→再执行→再验证，每阶段有门控 |
| **Context Rot** | 上下文腐化，Agent 在长周期任务中随着 token 增加导致推理质量下降的现象 |
| **Compaction** | 上下文压缩，将过长的对话历史压缩为摘要以防止 Context Rot 的策略 |
| **Garbage Collection Agent** | 垃圾回收 Agent，定期扫描代码库偏差、更新质量评分、提交重构 PR 的后台 Agent |
| **Test Ratchet** | 测试棘轮原则，禁止删除或修改失败的测试——只能让测试从 fail 变成 pass |
| **Harnessability** | 可驭缰性，技术/架构是否容易被 Harness 约束的程度——作为技术选型的第一类标准 |

### Harness Engineering 三层范式术语

| 术语 | 定义 |
|------|------|
| **Prompt Engineering** | 提示工程，优化单次提问措辞——解决"该怎么问" |
| **Context Engineering** | 上下文工程，管理模型在每一步看到的信息——解决"该让模型看到什么" |
| **Harness Engineering** | 驭缰工程，设计 Agent 运行的整个环境系统——解决"整个环境该怎么运作" |
| **三者关系** | 嵌套关系：Prompt ⊂ Context ⊂ Harness，每层包含前一层并解决更高级的失败模式 |

### Long-Running Agent 术语

| 术语 | 定义 |
|------|------|
| **Initializer Agent** | 初始化 Agent，在第一个 context window 中设置环境（init.sh、feature_list.json、初始 git commit） |
| **Coding Agent** | 编码 Agent，后续每个 session 增量推进一个 feature，留下 claude-progress.txt 进度追踪 |
| **Planner-Generator-Evaluator** | 三代理架构，类似GAN的生成-评估分离：Planner拆解任务→Generator实现→Evaluator验证 |
| **Sprint Contract** | Sprint合约，Evaluator和Generator在编码开始前协商"完成"标准 |
| **feature_list.json** | 功能清单JSON，Initializer生成的结构化任务列表（200+条目），比Markdown更不易被Agent误改 |
| **claude-progress.txt** | 进度追踪文件，Coding Agent在每个session结束时写入的交接备忘 |
| **Ralph Loop** | Ralph循环，Agent遇错自动迭代的持续循环机制——来自辛普森家族Ralph Wiggum角色 |
